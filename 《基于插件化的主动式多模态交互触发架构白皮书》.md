# 基于插件化的主动式多模态交互触发架构白皮书

> **实践基础**：本架构基于开源项目 **AiCode**（C++ 智能编码助手 CLI）工程实践，已在该项目中核心落地。
> 
> **项目地址**：https://github.com/ai-code/AiCode

---

## 1. 摘要

智能 Agent 发展至今，经历了四个关键阶段：

| 阶段 | 时间 | 特征 |
|------|------|------|
| 初探期 | 1956—1995 | 概念实体化，微软 Bob、Clippy 等产品尝试人机交互实践 |
| 分化期 | 1996—2011 | 产品类型化，软件 Agent 概念兴起但受限于硬件与算法 |
| 工程探索期 | 2012—2022 | 深度学习突破，Task/Planning 架构出现，Agent 工程化开始探索 |
| 标准化期 | 2023 至今 | 大模型驱动 Agent 爆发，但主动交互架构仍未成熟 |

**核心问题**：现有 Agent 设计普遍存在**架构缺陷**——触发逻辑与内核深度绑定、缺乏主动触发机制、插件扩展需修改源码、无生态闭环支撑。这些问题导致 Agent 功能固化、迭代困难，无法支撑规模化主动交互场景。

**本文创新**：提出一种**基于插件化的主动式多模态交互触发架构**，核心创新在于：
1. **插件化框架**：触发逻辑与调度内核彻底分离，支持插件热插拔、零代码扩展
2. **主动式交互**：业界首个主动交互触发框架，通过三模式分级触发实现"主动预判、分级低扰"的规模化交互
3. **生态闭环**：首个面向主动交互场景的插件生态——支撑"开发→审核→分发→反馈→迭代"的工程化体系（区别于现有 Skills 等被动 tool 型通用插件生态）

**工程实践基础**：本架构已在 **AiCode** 项目中落地验证——一个基于 C++ 开发的智能编码助手 CLI 工具，采用 REACT 范式（Think-Act-Observe 闭环），支持 Anthropic/Qwen/Ollama 多 LLM 提供者。

本架构不依赖大模型技术突破，而是从**架构设计层面**解决 Agent 主动交互的**工程化落地与规模化扩展**问题。

**核心创新聚焦**：
- **主动式交互**是核心价值主张
- **三模式分级触发**是实现主动式交互的技术手段（periodic/idle/idle_once）

## 2. 背景与痛点

### 2.1 传统 Agent 设计的核心缺陷

当前主流 Agent 设计存在以下**架构级问题**，制约了主动交互能力的实现：

| 缺陷 | 具体表现 | 导致的后果 |
|------|----------|-----------|
| **被动响应式架构** | 等待用户指令后触发 Agent Loop | 无法主动感知场景变化、预判用户需求 |
| **触发逻辑与内核绑定** | 触发条件硬编码在内核中 | 新增/修改触发规则需修改源码、重新编译 |
| **无事件分级机制** | 所有事件采用同一触发逻辑 | 无法兼顾紧急响应与用户免打扰 |
| **插件支持缺失** | 不支持多脚本格式，开发门槛高 | 功能扩展困难，插件复用率低 |
| **生态闭环缺失** | 无统一审核、分发、反馈渠道 | 插件孤立开发，无法形成迭代优化 |
| **无主动触发能力** | Agent Loop 仅支持被动调用 | 无法实现"预判需求、主动响应"的交互模式 |

> **关键洞察**：上述问题的根源不在于技术能力不足，而在于**架构设计缺陷**——触发逻辑与内核耦合、被动响应式架构、缺乏生态支撑。

### 2.2 本文架构的设计目标

针对传统 Agent 的架构缺陷，本文架构的设计目标如下：

| 目标 | 对应解决的传统 Agent 缺陷 |
|------|-------------------------|
| **架构解耦** | 触发逻辑与内核绑定，扩展需修改源码 |
| **主动触发** | 被动响应式架构，无法预判用户需求 |
| **分级交互** | 无事件分级，无法平衡响应与免打扰 |
| **灵活调用** | 插件支持缺失，开发门槛高 |
| **生态闭环** | 无统一生态支撑，无法迭代优化 |

## 3. 相关工作（Related Work）

### 3.1 被动响应式 Agent 发展脉络

| 阶段 | 代表产品/框架 | 触发方式 | 局限性 |
|------|------------|---------|--------|
| 早期聊天机器人（1966-2010） | ELIZA、Alice | 用户输入 → 规则匹配 → 回复 | 无主动交互能力 |
| 智能助手（2011-2022） | Siri、Alexa、小爱同学 | 唤醒词 → 语音识别 → 执行 | 需用户主动唤醒，无法预判 |
| 大模型 Agent（2023 至今） | ChatGPT、Claude、OpenAI Assistants API | 用户指令 → Agent Loop → 工具调用 | **Agent Loop 仅支持被动调用**，无主动触发机制 |
| 插件生态（2023 至今） | OpenAI Skills、Claude MCP | 用户/Agent 主动调用插件 | **被动 tool 型插件**，需等待调用 |

> **关键洞察**：现有 Agent 架构（包括大模型 Agent）的触发机制均为**被动响应式**——等待用户指令后启动 Agent Loop。

### 3.2 规则引擎式主动交互工具

| 产品 | 领域 | 触发方式 | 局限性 |
|------|------|---------|--------|
| Hazel（macOS） | 文件自动化 | 监听文件变化 → 执行预设动作 | 固定规则，无法扩展，无智能响应 |
| AutoHotkey（Windows） | 桌面自动化 | 热键/事件 → 执行脚本 | 脚本与配置绑定，无插件生态 |
| IFTTT / Zapier | Web 服务自动化 | Webhook/定时 → 调用 API | 云端规则引擎，非本地 Agent 架构 |
| Home Assistant | 智能家居 | 传感器状态 → 联动设备 | 封闭规则配置，非插件化 |

> **关键洞察**：此类工具实现了**基于事件的主动触发**，但采用**固定规则引擎**模式，无法通过插件扩展，且无大模型联动能力。

## 4. 核心设计理念

以"**架构解耦为核心、主动触发为突破、分级低扰为支撑、生态闭环为保障**"，通过架构级别创新重构 Agent 主动交互触发体系。

**核心理念对比**：

| 维度 | 传统 Agent 设计 | 本文架构设计 |
|------|---------------|-------------|
| 架构模式 | 单体耦合 | 三层解耦 |
| 触发方式 | 被动响应（用户指令驱动） | 主动预判（事件/状态驱动） |
| 扩展方式 | 修改源码 + 重新编译 | 插件热插拔 + 零内核改动 |
| 事件处理 | 统一处理，无分级 | 三模式分级（强制/空闲/单次） |
| 生态支撑 | 无/零散 | 插件社区闭环 |
| 迭代能力 | 依赖版本更新 | 插件自主迭代 + 反馈闭环 |

## 5. 架构整体设计

本文架构分为**运行时架构**和**生态支撑体系**两部分：
- **运行时架构**（两层创新 + 一层复用）：
  - **插件层**（创新）：触发逻辑与内核解耦，支持热插拔
  - **调度层**（创新）：三模式分级触发，主动感知与调度
  - **执行层**（复用）：复用现有 Agent 框架的工具执行能力（ReAct Act 部分）
- **生态支撑体系**：插件社区（服务端 + 客户端）—— 独立基础设施

### 5.2 生态支撑体系：插件社区

插件社区是独立于运行时架构的基础设施，为架构提供生态支撑：

| 组成 | 功能 |
|------|------|
| **服务端** | 插件上传、安全检测、人工审核、分类存储、版本管理、数据统计 |
| **客户端**（运行时内嵌） | 插件检索、一键下载、自动部署、版本更新、反馈提交 |
| **价值** | 构建"开发→审核→分发→使用→反馈→迭代"的完整生态闭环 |

**差异化定位**：

| 生态类型 | 代表产品 | 服务对象 | 插件类型 |
|---------|---------|---------|---------|
| 通用插件生态 | OpenAI Skills、Claude MCP | 被动响应式 Agent | 被动 tool 型插件 |
| **主动交互插件生态** | 本文插件社区 | **主动式交互 Agent** | **触发规则型插件** |

> **说明**：业内已有通用插件生态（如 Skills、MCP），但它们是**被动 tool 型**插件，需等待用户或 Agent 主动调用。本文插件社区专注于**主动触发规则型**插件，填补主动交互领域空白。

### 5.3 运行时架构各层说明

#### 5.3.1 插件层（业务逻辑层）

**传统 Agent 问题**：触发逻辑硬编码在内核中，扩展需修改源码。

**本文创新**：
- **存储目录**：全局统一根目录 `active/`，所有插件为该目录下独立子文件夹
- **插件组成**：每个插件必须包含 3 个核心文件：
  - `trigger`（触发脚本，任意语言，执行返回 true/false）
  - `trigger_mode.cfg`（模式配置文件：periodic/idle/idle_once）
  - `ACTIVE.md`（交互配置文档，用于大模型联动）

**插件示例**（CPU 温度检测插件）：

```
active/
└── cpu_temperature_monitor/
    ├── trigger          # 触发脚本（任意语言，返回 true/false）
    ├── trigger_mode.cfg # 模式配置
    └── ACTIVE.md        # 交互配置
```

**trigger 触发逻辑流程图**：

```
┌─────────────────────────────────────────────────────────────┐
│  trigger 执行流程                                            │
├─────────────────────────────────────────────────────────────┤
│  1. 执行检测逻辑（如读取 CPU 温度）                            │
│          │                                                  │
│          ▼                                                  │
│  2. 判断是否达到触发条件                                    │
│          │                                                  │
│     ┌────┴────┐                                             │
│     │ Yes     │ No                                          │
│     ▼         ▼                                             │
│  返回 true   返回 false                                      │
│  表示触发    表示不触发                                      │
│          │                                                  │
│          ▼                                                  │
│  3. 输出触发原因到 stdout（可选）                            │
│     例："CPU 温度 90°C，超过阈值 85°C"                        │
└─────────────────────────────────────────────────────────────┘
```

**输出规范**：
- 返回 `true` = 触发（可打印触发原因到 stdout）
- 返回 `false` = 不触发
- 超时 `100ms` 强制终止，避免阻塞调度

**trigger_mode.cfg 配置说明**：

| 配置项 | 说明 | 示例值 |
|--------|------|--------|
| mode | 触发模式 | periodic（周期性）/ idle（空闲触发）/ idle_once（单次空闲） |
| interval | 执行间隔（秒） | 60 |
| priority | 优先级 | high（高）/ low（低） |

**ACTIVE.md 配置说明**：

| 字段 | 说明 |
|------|------|
| 触发场景 | 描述触发条件和事件类型 |
| 交互话术 | 大模型生成回调的参考模板 |
| 可执行操作 | 用户可一键执行的任务列表 |

**触发后流程**：

```
trigger 返回 true → 读取 ACTIVE.md → 发送给大模型 → 生成响应 → 通知用户/执行任务
```

- **插件特性**：支持热插拔，新增/修改/卸载插件无需重启架构，无需修改内核源码

#### 5.3.2 调度层（核心中枢层）

**传统 Agent 问题**：调度逻辑与触发逻辑耦合，无法灵活适配不同触发场景。

**本文创新**：
- **插件扫描与加载**：遍历插件目录，校验插件完整性，分类存入对应插件池；支持热加载，新增/删除插件无需重启
- **三模式分级调度**：按触发类型（periodic/idle/idle_once）差异化调用
- **脚本管控与结果解析**：子进程隔离调用，严格解析触发状态（true=触发，false=不触发）
- **优先级调度**：强制类插件优先于空闲类插件，解决并发冲突
- **大模型联动**：触发成功后，读取 ACTIVE.md 并发送给大模型，接收回调响应并下发至执行层

#### 5.3.3 执行层（交互落地层）

**说明**：执行层复用现有 Agent 框架的工具执行能力（如 ReAct 的 Act 部分），本文不作创新。

**本文聚焦**：
- **插件层**：触发逻辑与内核解耦，支持插件化扩展
- **调度层**：三模式分级触发，实现主动式交互
- **生态支撑**：插件社区闭环，支撑规模化工程化落地

## 6. 核心功能实现

### 6.1 主动式交互实现：三模式分级触发

**主动式交互核心理念**：从"被动响应"升级为"主动预判"。

**与"持续采样输入的主动交互"的区别**：

| 维度 | 持续采样输入（如文件监控、Hazel） | 本文架构 |
|------|---------------------------------|---------|
| 定位 | **感知技术手段**：轮询/监听系统状态 | **工程化框架**：容纳各种感知方式 |
| 触发逻辑 | 固定规则：if 变化 → then 动作 | 插件化：trigger 返回 true/false |
| 响应方式 | 执行预设动作 | 读取 ACTIVE.md → 大模型生成响应 → 通知/执行 |
| 扩展方式 | 需修改代码/配置规则 | 新增插件，无需改内核 |
| 生态支撑 | 无，孤立使用 | 插件社区闭环 |
| 迭代能力 | 无，规则固化 | 插件可更新、可反馈迭代 |

> **说明**：持续采样输入（如监听文件、鼠标、系统状态）是一种**感知技术手段**，可作为本文插件的实现方式。但本文架构的核心价值在于提供**主动交互的工程化框架**——感知方式由插件决定，触发后联动大模型生成智能响应，并通过插件生态实现规模化扩展。

**本文聚焦**：
- **插件化框架**：触发逻辑与内核解耦，支持任意感知方式
- **大模型联动**：触发后读取 ACTIVE.md，生成场景化响应
- **生态闭环**：插件社区支撑"开发→审核→分发→反馈→迭代"

**交互主体性分析**：用户预设触发规则，系统主动发起交互

| 维度 | 传统被动交互 | 本文架构交互 |
|------|------------|-------------|
| 触发主体 | 用户指令 | 插件脚本（用户意志延伸） |
| 交互发起方 | 用户 | 系统（Agent） |
| 示例 | 用户输入 `/help` → Agent 响应 | trigger 返回 true → Agent 主动通知用户 |

**交互流程**：
```
用户预设阶段：
  用户下载/编写插件 → 注册到 active/ 目录 → 配置触发条件
  ↓
  运行时阶段：
  调度器扫描插件 → trigger 脚本返回 true → 大模型生成响应 → 主动通知用户
```

> **说明**：这是**用户预设的系统主动交互**，而非大模型"自主意识"式的主动交互。插件脚本是用户意志的延伸，大模型的"主动"体现在：在满足用户预设条件时，**无需用户再次指令**即生成响应。类比手机闹钟——用户设置闹钟，闹钟响时主动提醒用户。

### 6.1.1 主动交互层级定位

本文架构的"主动交互"属于**工程化可落地的条件主动**，区别于科幻式的"AI 自主意识"主动交互。

**主动交互层级**：

```
┌─────────────────────────────────────────────────────────────┐
│  主动性层级                                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ① 完全被动 ← ChatGPT 等传统 Agent：等待用户指令              │
│                                                              │
│  ② 条件主动 ← 本文架构：用户预设条件，系统主动发起            │
│                                                              │
│  ③ 自主主动 ← 科幻式 AI：Agent 自主感知、自主决策、主动服务   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**层级对比**：

| 维度 | ① 完全被动 | ② 条件主动 (本文) | ③ 自主主动 |
|------|-----------|-----------------|-----------|
| 交互发起时机 | 用户指令 | 条件满足 | Agent 自主判断 |
| 触发决策主体 | 用户 | 用户预设 + 系统执行 | Agent 自主 |
| 响应内容生成 | Agent 生成 | Agent 生成 | Agent 生成 |
| 主动性程度 | 完全被动 | 条件主动 | 完全主动 |
| 工程可行性 | ✅ 成熟 | ✅ 可落地 | ⚠️ 技术未成熟 |

**本文定位**：
- ✅ **交互时机主动**：无需用户发起指令
- ✅ **响应生成主动**：大模型生成场景化回复
- ⚠️ **触发条件预设**：用户定义关切点，非 Agent 自主判断

> **边界说明**：本文架构是**工程化可落地的 Agent 主动交互**，在"用户预设条件"与"系统主动执行"之间取得平衡，避免过度解读为"AI 自主意识"。

**三模式分级触发机制**（对比传统 Agent）：

| 维度 | 传统 Agent | 本文架构 |
|------|-----------|---------|
| 触发模式 | 单一：仅支持用户指令触发 | 三模式：periodic/idle/idle_once |
| 事件分级 | 无分级，统一处理 | 分级：严重级（强制）+ 普通级（空闲） |
| 免打扰支持 | 不支持 | 支持：用户交互时中断普通触发 |
| 单次触发 | 无 | 支持：idle_once（一次性引导） |
| 超时保护 | 无 | 支持：脚本超时 100ms，避免阻塞 |

**三类触发模式**：

| 模式 | 触发条件 | 适用场景 | 保障机制 |
|------|----------|----------|----------|
| **周期性强制触发**<br>（periodic） | 按配置周期定时触发 | 严重级事件（硬件故障等） | 脚本超时 100ms，避免阻塞调度 |
| **用户空闲触发**<br>（idle） | 与大模型无交互达到阈值时触发 | 普通级事件（空闲提醒等） | 用户恢复交互时立即中断 |
| **单次空闲触发**<br>（idle_once） | 与大模型无交互达到阈值时触发一次 | 一次性建议（如首次空闲引导） | 触发后自动禁用，需手动重置 |

**触发模式关系图**：

```
┌──────────────────────────────────────────────────────────────┐
│                    触发模式分类体系                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  触发模式                                                     │
│      │                                                        │
│      ├─────────────────┬─────────────────┐                    │
│      │                 │                 │                    │
│      ▼                 ▼                 ▼                    │
│  强制类             空闲类          外部触发类                │
│  (不受用户状态影响)   (依赖用户状态)    (预留扩展)              │
│      │                 │                 │                    │
│      │                 ├──────┬──────┐   ▼                    │
│      │                 │      │      │  Webhook             │
│      ▼                 ▼      ▼      │  (API 调用触发)        │
│   periodic           idle  idle_once  │                       │
│   (周期触发)         (空闲) (单次空闲) │                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**触发模式说明**：

| 类别 | 模式 | 触发条件 | 触发次数 |
|------|------|----------|----------|
| 强制类 | periodic | 按 interval 周期定时 | 无限次 |
| 空闲类 | idle | 与大模型无交互 ≥ threshold | 空闲期间多次 |
| 空闲类 | idle_once | 与大模型无交互 ≥ threshold | 仅一次 |
| 外部类 | Webhook | API 调用触发 | 按需 |

**配置示例**（`trigger_mode.cfg`）：

```ini
# ==========================================
# 周期性强制触发（严重级事件）
# ==========================================
mode=periodic
interval=60        # 每 60 秒触发一次
priority=10        # 优先级：0-100，数字越小优先级越高
timeout=100        # 脚本超时 100ms

# ==========================================
# 用户空闲触发（普通级事件）
# ==========================================
mode=idle
threshold=300      # 与大模型无交互 300 秒后触发
interval=60        # 进入空闲模式后，每 60 秒执行一次
priority=50        # 中等优先级
timeout=100

# ==========================================
# 单次空闲触发（一次性空闲引导）
# ==========================================
mode=idle_once
threshold=120      # 与大模型无交互 120 秒后触发
priority=40        # 中低优先级
timeout=100
# 注意：trigger 返回 true 执行一次后，插件自动禁用
# 需手动重置状态或删除插件重新启用
```

**优先级说明**：

| 优先级 | 数值范围 | 说明 | 示例场景 |
|--------|---------|------|---------|
| 紧急 | 0-10 | 系统级告警，立即执行，可中断用户操作 | 硬件故障、安全威胁 |
| 高 | 11-30 | 重要通知，优先调度 | 关键任务提醒 |
| 中高 | 31-50 | 普通提醒，用户空闲时执行 | 文件整理、缓存清理 |
| 中 | 51-70 | 建议类通知，低干扰 | 使用技巧推荐 |
| 低 | 71-100 | 后台任务，最低优先级 | 数据同步、日志归档 |

### 6.1.2 多周期独立调度器

调度层采用**多周期独立调度策略**，为每个 periodic 插件按配置的 `interval` 独立定时触发，实现精准调度与低空转。

**核心设计**：

```
┌──────────────────────────────────────────────────────────────┐
│                  多周期独立调度示意图                         │
├──────────────────────────────────────────────────────────────┤
│                                                               │

│  ┌─────────────┐                                             │
│  │ Plugin A    │──► 每 10 秒触发一次                          │
│  │ interval=10 │                                             │
│  └─────────────┘                                             │
│                                                               │
│  ┌─────────────┐                                             │
│  │ Plugin B    │──► 每 60 秒触发一次                          │
│  │ interval=60 │                                             │
│  └─────────────┘                                             │
│                                                               │
│  ┌─────────────┐                                             │
│  │ Plugin C    │──► 每 300 秒触发一次                         │
│  │ interval=300│                                             │
│  └─────────────┘                                             │
│                                                               │
│  调度器：计算最早到期时间 → 睡眠 → 唤醒执行 → 循环            │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**调度策略对比**：

| 策略 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| 统一扫描周期 | 固定周期（如每 10 秒）扫描所有插件，检查是否到期 | 实现简单 | 空转浪费 CPU，调度精度低 |
| **独立周期触发** | 每个插件按自己的 `interval` 独立定时触发 | 精准调度、低空转 | 需要定时器管理 |

**本文采用独立周期触发策略**，优势如下：

```
┌──────────────────────────────────────────────────────────────┐
│  独立周期调度优势                                            │
├──────────────────────────────────────────────────────────────┤

│                                                              │
│  ✅ 低空转：定时器睡眠唤醒机制，不占用 CPU                    │
│                                                              │
│  ✅ 高扩展：新增插件动态加入调度队列，无需重启                │

│  ✅ 优先级保障：紧急插件可优先调度执行                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.1.3 空闲触发状态机

空闲触发采用**三阶段状态机**设计，确保"用户专注不打扰，空闲时刻主动服务"。

**空闲定义**：本架构中的"用户空闲"指**用户与大模型无交互**，包括：
- 无对话请求（用户未发送新消息）
- 无任务执行（用户未触发 Agent 任务）
- 无反馈操作（用户未对大模型回复进行评价/操作）

> **说明**：采用"与大模型交互"而非"系统输入事件"作为空闲判断依据，避免采集鼠标/键盘等隐私数据，同时更精准反映用户对 Agent 的注意力状态。

**状态机说明**：

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|----------|----------|
| ① 监测中 | 监测用户与大模型交互事件 | 插件启动/用户恢复交互 | 与大模型无交互 ≥ threshold |
| ② 待触发 | 标记为可触发，等待调度 | 与大模型无交互 ≥ threshold | 调度层调用 |
| ③ 触发中 | 按 interval 周期执行 trigger | 调度层调用 | 用户恢复交互 |

**配置示例**（`trigger_mode.cfg`）：

```ini
# 用户空闲触发（增强版配置）
mode=idle
threshold=300      # 与大模型无交互 300 秒后，进入空闲触发模式
interval=60        # 进入空闲模式后，每 60 秒执行一次 trigger 脚本
priority=50        # 优先级
timeout=100        # 脚本超时 100ms
```

**交互事件类型**：

| 事件类型 | 说明 | 示例 |
|----------|------|------|
| 对话请求 | 用户主动发送消息 | 用户输入"/help" |
| 任务执行 | 用户触发 Agent 任务 | 用户点击"执行"按钮 |
| 反馈操作 | 用户对回复进行操作 | 点赞/点踩/复制回复 |

**双参数独立控制的优势**：

| 参数 | 控制对象 | 调整效果 |
|------|----------|----------|
| `threshold` | 进入空闲模式的门槛 | 越大越少触发，越小越敏感 |
| `interval` | 空闲模式内的执行频率 | 越大执行越少，越小检测越频繁 |

```
┌──────────────────────────────────────────────────────────────┐
│  配置组合示例                                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  场景 1：轻度打扰（如建议类通知）                             │
│  threshold=600, interval=300                                 │
│  → 与大模型无交互 10 分钟触发，触发后每 5 分钟检测一次         │
│                                                              │
│  场景 2：中度打扰（如文件整理提醒）                           │
│  threshold=300, interval=120                                 │
│  → 与大模型无交互 5 分钟触发，触发后每 2 分钟检测一次          │
│                                                              │
│  场景 3：重度打扰（如久坐提醒）                               │
│  threshold=120, interval=60                                  │
│  → 与大模型无交互 2 分钟触发，触发后每分钟检测一次             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.1.4 单次空闲触发模式（idle_once）

在周期性和空闲触发基础上，增加**单次空闲触发**变体，满足"一次性引导"场景需求。

**核心思想**：与大模型无交互达到阈值时触发一次，之后自动禁用。

**适用场景**：

| 场景 | 说明 | 示例 |
|------|------|------|
| 新手引导 | 首次与大模型无交互时提供引导 | "首次使用？点击查看功能介绍" |
| 一次性建议 | 检测到特定状态后建议一次 | "检测到大量未整理文件，是否整理？" |
| 限时提醒 | 触发后用户未响应则不再提醒 | "已空闲 2 小时，建议休息" |

**空闲定义**：同 6.1.3 节，指**用户与大模型无交互**（无对话请求、无任务执行、无反馈操作）。

**状态机说明**：

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|----------|----------|
| ① 监测中 | 监测用户与大模型交互事件 | 插件启动 | 与大模型无交互 ≥ threshold |
| ② 待触发 | 标记为可触发，等待调度 | 与大模型无交互 ≥ threshold | 调度层调用 |
| ③ 触发中 | 执行 trigger，返回 true 则执行并禁用 | 调度层调用 | trigger 返回 true → 已禁用 |

**idle_once 与 idle 的区别**：

| 维度 | idle（空闲周期） | idle_once（单次空闲） |
|------|-----------------|----------------------|
| 触发次数 | 空闲期间按 interval 多次触发 | 仅触发一次 |
| 触发后状态 | 继续监测，用户恢复交互后重置 | 自动禁用，需手动重置 |
| 典型场景 | 持续建议（如缓存清理） | 一次性引导（如新手提示） |

**手动重置方式**：

| 方式 | 说明 |
|------|------|
| 删除插件目录后重新下载 | 最彻底的重置方式 |
| 修改 trigger_mode.cfg 中 mode 字段 | 快速重置 |
| 通过管理命令重置 | 如 `/active reset <plugin_name>` |
| 插件内实现自复位逻辑 | 检测特定文件后重置 |

### 6.1.5 三种触发模式对比

| 模式 | 触发前提 | 触发次数 | 适用场景 | 配置复杂度 |
|------|----------|----------|----------|------------|
| **periodic** | 无，按 interval 定时 | 无限次 | 持续监控（CPU、内存） | 低 |
| **idle** | 与大模型无交互 ≥ threshold | 空闲期间多次 | 空闲建议（缓存清理） | 中 |
| **idle_once** | 与大模型无交互 ≥ threshold | 仅一次 | 一次性引导（新手提示） | 低 |

**选择建议**：

**快捷配置**：

| 配置方式 | 说明 | 示例 |
|----------|------|------|
| `threshold=0` | 跳过空闲检测，直接周期性触发 | `mode=idle, threshold=0` → 等价于 periodic |
| `threshold>0` | 启用空闲检测 | `mode=idle, threshold=300` |
| `mode=idle_once` | 单次空闲触发，自动禁用 | `threshold=120` |

> **说明**：`idle` 模式下 `threshold=0` 时，空闲检测自动跳过，等价于 `periodic` 模式，无需手动选择触发模式。

### 6.2 插件化实现（对比传统方案）

**传统方案无主动式插件化设计**：传统工具（Hazel、AutoHotkey、文件监控等）采用**固定规则引擎**模式，规则与程序绑定，无法通过插件形式扩展主动触发能力。

| 维度 | 传统方案（Hazel、AutoHotkey 等） | 本文架构 |
|------|-------------------------------|---------|
| 架构模式 | 固定规则引擎（if-then 配置） | 主动式插件化框架 |
| 触发器 | 无标准化格式，规则写死在配置中 | 标准化 trigger 脚本（返回 true/false） |
| 扩展方式 | 修改配置或编写新脚本 | 新增插件，无需改内核 |
| 响应方式 | 执行预设动作 | 大模型生成场景化响应 |
| 加载方式 | 重启生效 | 热加载（30 秒自动检测） |
| 生态支撑 | 无，孤立使用 | 插件社区闭环 |

**本文核心价值**：首个面向**主动交互场景**的插件化框架，将零散的主动触发规则升级为可共享、可迭代的工程化体系。

### 6.3 大模型联动实现

**触发后核心流程**：

```
trigger 返回 true → 读取 ACTIVE.md → 构建请求 → 调用大模型 → 解析响应 → 通知/执行
```

**详细流程**：

```

│     可选输出触发原因到 stdout                                │
│          │                                                  │
│          ▼                                                  │
│  2. 读取 ACTIVE.md                                           │
│     - 触发场景描述                                          │
│     - 交互配置（含模式类型）                                 │
│          │                                                  │
│          ▼                                                  │
│  3. 创建临时会话，调用 Agent Loop                           │
│          │                                                  │
│          ▼                                                  │

│          │                                                  │
│     ┌────┴────────────────┬────────────────────────────┐    │
│     ▼                     ▼                            ▼    │
│  类型 A：主动交互    类型 B：执行任务 + 通知       类型 C：静默执行   │

```

**说明**：A/B/C 三类为典型的主动交互模式示例，本架构通过插件化设计支持无限扩展更多交互模式。

**响应类型示例**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **类型 A：主动交互（对话 + 情绪）** | 大模型生成回复内容，与用户对话，支持情绪交流 | "CPU 温度过高，建议检查散热风扇" / "工作再忙也要记得休息哦" |
| **类型 B：执行任务 + 通知（任务编排）** | Agent 根据 ACTIVE.md 编排任务，完成后总结通知 | "已清理 5 个后台进程，CPU 温度下降 3°C" |
| **类型 C：静默执行（后台例行）** | Agent 静默执行任务，不通知用户 | 后台缓存清理 → 无感知完成 |

---

### 6.3.1 类型 A：主动交互（对话 + 情绪）

- **本质**：大模型作为"对话者"，生成回复内容与用户交互，支持情绪交流
- **流程**：触发 → 创建临时会话 → 大模型生成回复 → UI 展示 → 销毁
- **特征**：轻量级、非阻塞、支持情绪交流（关心、提醒、鼓励等）
- **场景**：告警、建议、提醒、情感陪伴、一次性指导

### 6.3.2 类型 B：执行任务 + 通知（任务编排）

- **本质**：Agent 作为"执行者"，根据 ACTIVE.md 编排任务并执行
- **流程**：触发 → 创建临时会话 → Agent 通过 ToolRegistry 执行 → 总结通知 → 销毁
- **特征**：完整工具调用链、任务编排、渐进式通知、结果可量化
- **场景**：需用户知晓的任务（系统清理、文件归档、数据备份等）

### 6.3.3 类型 C：静默执行（后台例行）

- **本质**：Agent 作为"后台服务"，无感知完成任务
- **流程**：触发 → 创建临时会话 → Agent 执行 → 不通知 → 可选日志 → 销毁
- **特征**：零打扰、后台执行、可选日志记录供事后审计
- **场景**：定时清理、数据同步、日志轮转、健康检查等例行任务

---

### 6.3.4 三模式对比

| 维度 | 类型 A（对话） | 类型 B（任务） | 类型 C（静默） |
|------|--------------|--------------|---------------|
| **Agent 角色** | 对话者/助手 | 执行者/管家 | 后台服务 |
| **工具调用** | 否 | 是 | 是 |
| **用户通知** | 是（交互/情绪） | 是（结果总结） | 否 |
| **用户感知** | 强感知 | 中等感知 | 无感知 |
| **典型场景** | 告警/建议/陪伴 | 需知晓的任务 | 后台例行任务 |

**模式选择原则**：
- **类型 A**：用户需要知情或决策，或需要情绪价值
- **类型 B**：任务需要执行且用户需要知晓结果
- **类型 C**：例行任务，无需打扰用户

### 6.4 插件生态闭环实现

**传统 Agent 问题**：无生态支撑，插件孤立开发，无法形成迭代优化。

**本文核心支撑**：
- **服务端**：插件上传、安全审核、分类管理、版本迭代、反馈收集
- **客户端**：插件检索、一键下载、自动更新、本地管理、反馈提交
- **离线适配**：支持离线使用本地插件，网络恢复后自动同步数据

## 7. 主动交互价值与应用场景

### 7.1 核心价值：从传统 Agent 到本文架构的三大突破

| 突破 | 从...到... | 说明 |
|------|-----------|------|
| 1 | **被动响应 → 主动预判** | 无需用户主动发起指令，系统自主感知场景变化、预判需求 |
| 2 | **零散触发 → 闭环迭代** | 结合 Agent Loop，记录执行结果与用户反馈，完成迭代优化 |
| 3 | **固定功能 → 插件扩展** | 插件化框架 + 生态闭环，支持规模化业务场景扩展 |

### 7.2 典型应用场景

| 场景 | 应用方式 | 传统 Agent 局限 | 本文架构优势 |
|------|----------|---------------|-------------|

## 8. 总结与展望

### 8.1 本文创新总结

本架构的核心突破在于**架构级别创新**，针对传统 Agent 设计的核心缺陷提出系统性解决方案：

| 传统 Agent 缺陷 | 本文创新方案 | 实现效果 |
|---------------|-------------|---------|
| 触发逻辑与内核绑定 | 插件化框架，触发逻辑下沉至插件层 | 插件热插拔，零内核改动扩展 |
| 被动响应式架构 | **主动式交互框架**（三模式分级触发实现） | 主动预判，紧急响应与免打扰兼顾 |
| 无生态闭环支撑 | 插件社区（开发→审核→分发→反馈） | 可持续迭代的工程化体系 |
| 扩展需修改源码 | 标准化插件规范（3 核心文件） | 低门槛开发，规模化扩展 |

**核心创新点**：
1. **主动式交互**（核心价值）：业界首个主动交互触发框架 —— 从"被动响应"升级为"主动预判"。支持三种典型交互模式（通过插件可扩展更多）：
   - **类型 A：主动交互（对话 + 情绪）**：大模型生成回复内容，与用户对话，支持情绪交流（如"CPU 温度过高，建议检查散热风扇" / "工作再忙也要记得休息哦"）
   - **类型 B：执行任务 + 通知（任务编排）**：Agent 根据 ACTIVE.md 编排任务，完成后总结通知（如"已清理 5 个后台进程，CPU 温度下降 3°C"）
   - **类型 C：静默执行（后台例行）**：Agent 静默执行任务，不通知用户（如后台缓存清理 → 无感知完成）
2. **插件化框架**（架构支撑）：触发逻辑与内核解耦 + 插件热插拔 —— 解决"触发逻辑与内核绑定"的行业共性痛点
3. **工程化设计**（规模化保障）：插件社区闭环 + 标准化插件规范 —— 支撑规模化扩展与可持续迭代
4. **三模式触发**（技术实现）：periodic/idle/idle_once 三种触发模式，覆盖持续监控、空闲建议、一次性引导全场景

### 8.2 展望

未来将围绕架构创新持续优化，重点推进三个方向：

1. **标准化推动**：推动主动交互插件规范成为行业标准
2. **场景库建设**：积累主动触发场景模板，降低插件开发门槛
3. **active 插件生态建设**：构建主动交互插件社区，支撑"开发→审核→分发→反馈→迭代"闭环
